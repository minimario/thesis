Most of today's benchmarks measure the ability of language models to solve
a single, static programming task. However, this does not adequately capture
the way that software is developed in the real world. Codebases frequently
evolve over time as new features are added, requirements change, and designs
are prototyped. The design decisions and abstractions introduced in a codebase
frequently have long-term consequences for the maintainability and extensibility
of the codebase.


SlopCodeBench was designed to capture this setting and desiderata. It contains
\transProblems{} language-agnostic problems spanning
\overallTotalCheckpoints{} checkpoints. Each problem is specified solely in
terms of observable behavior at a command-line or API boundary. At the first
checkpoint, an agent implements the initial specification from scratch; at each
later checkpoint, it must modify and extend its own prior solution. Figure \ref{fig:slopcodebench-overview} illustrates this process.

    \caption{Iterative evaluation in SlopCodeBench. A task unfolds over
    checkpoints. At each step, the agent extends its prior solution to satisfy
    an updated specification. Correctness, erosion, and verbosity are scored
    externally and withheld from the agent.}

\end{figure}

Below is an example of a task from SlopCodeBench:
\begin{runningexample}[Example: \href{https://www.scbench.ai/problems/code_search}{\codesearch{}}]
\label{box:slopcodebench-code-search}
\textbf{Running example: \codesearch{}.}
The agent is tasked with building a CLI tool for semantic source-file search,
inspired by \href{https://ast-grep.github.io}{ast-grep}, across five checkpoints:
\begin{itemize}
    \item[$\ckpt{1}$ -] Python files only exact and regex matching. Establishes the core CLI contract.
    \item[$\ckpt{2}$ -] Support for Javascript and C++.
    \item[$\ckpt{3}$ -] AST-based pattern matching with metavariable capture.
    \item[$\ckpt{4}$ -] Selector rules and auto-fix functionality.
    \item[$\ckpt{5}$ -] Add support for Go, Rust, and Java.
\end{itemize}
An agent that hardcodes language-specific logic at $\ckpt{1}$ faces cascading
rewrites at $\ckpt{2}$ and $\ckpt{5}$; one that builds an extensible parser
interface does not.
\end{runningexample}

In addition to an iterative benchmark, SlopCodeBench introduced two trajectory-level metrics to measure the quality of
the generated code as it evolves: \textbf{structural erosion} and \textbf{verbosity}.
Structural erosion captures the concentration of complexity in already-complex functions.
Verbosity captures how much redundant, duplicated, or unnecessary code is present.
These metrics are designed to measure two dimensions of code quality while being independent of
functional correctness.

\subsection{Measuring Code Slop}
The central metric contribution is to measure not only whether the code passes
tests, but how its quality changes over time. SlopCodeBench tracks two
trajectory-level quality signals: \textbf{structural erosion} and
\textbf{verbosity}.

Structural erosion captures the concentration of control-flow complexity in
already-complex functions. We first scored each callable $f$ with a
\emph{complexity mass}. In this formula, $\text{CC}(f)$ is the cyclomatic
complexity of $f$: roughly, the number of independent paths through the
callable's control flow, increasing with branches, loops, and other decision
points. $\text{SLOC}(f)$ is the number of source lines of code in $f$. We defined
\begin{equation*}
\label{eq:slopcodebench-mass}
\text{mass}(f) = \text{CC}(f) \times \sqrt{\text{SLOC}(f)}.
\end{equation*}

Erosion measures the fraction of the codebase's total complexity mass that sits
inside high-complexity functions:
\begin{equation}
\label{eq:slopcodebench-erosion}
\text{Erosion}
=
\frac{\sum_{f \in \mathcal{F}} \mathbb{I}[\text{CC}(f) > 10]\cdot\text{mass}(f)}
     {\sum_{f \in \mathcal{F}} \text{mass}(f)}.
\end{equation}
Here, $\mathcal{F}$ is the set of callables in the program, and
$\mathbb{I}[\text{CC}(f) > 10]$ is an indicator that equals 1 only for callables
whose cyclomatic complexity exceeds 10. This metric is designed to penalize an anti-pattern where
instead of introducing helpful abstractions, an agent repeatedly patches a large dispatch function
or main loop until the codebase's complexity collapses into a few hard-to-maintain functions.

Verbosity captures a different failure mode: redundant, duplicated, or
unnecessary code. It combines two sources of evidence. The first is lines
flagged by 137 AST-Grep rules for wasteful patterns; the second is lines that
belong to structural code clones. We deduplicate overlapping hits and normalize
by $\text{LOC}$, the total lines of code in the solution:
\begin{equation*}
\label{eq:slopcodebench-verbosity}
\text{Verbosity}
=
\frac{|\{\text{AST-Grep Flagged Lines} \cup \text{Clone Lines}\}|}
     {\text{LOC}}.
\end{equation*}

\subsection{Measuring Correctness and Cost Over Time}
In Table~\ref{tab:performance-overview}, we showed the performance of models
on SlopCodeBench. We grouped solutions into three categories: strict, isolated, and core.
Core means that a solution passes only a set of core tests, testing
behavior explicitly mentioned in the specification. Isolated (Iso) means that
the solution passes all tests that only depend on the current checkpoint (it can fail regression tests).
Strict means that the solution passes the full test suite, including regression tests from prior checkpoints.

\subsection{Measuring Code Quality Over Time}
Next, we examined the quality of code generated by agents throughout the process
of completing checkpoints for each task. As shown in Figure~\ref{fig:slopcodebench-quality-accumulation},
both structural erosion and verbosity tended to increase as agents progressed through the checkpoints.
Across all settings, structural erosion increased in
\erosionPctIncreasing{}\% of trajectories and verbosity increased in
\vfTrajIncPct{}\%. The average number of functions with cyclomatic complexity
at least 10 rose from \erosionHighCcCountFirstMean{} to
\erosionHighCcCountLastMean{}, and mean maximum cyclomatic complexity rose
from \newErosionFirstCcMaxMean{} to \newErosionLastCcMaxMean{}.

\subsection{Calibration Against Human Repository Histories}
Finally, we compared agent trajectories against existing human-maintained Python

other domains, with up to 30 source-modifying commits per repository for a

\subsection{Discussion}
Overall, SlopCodeBench shed light on a model's ability to write high-quality code in an environment
that mirrors real-world software development. Rather than testing one-shot performance,
our iterative feature-adding paradigm requires that the models plan ahead and write code
that can be maintained through many iterations. Our benchmark made evident that as more feature requirements were added, models struggled
to maintain correctness despite increasing cost. In addition, by introducing verbosity and structural erosion metrics,
we were able to quantitatively observe that code quality degraded throughout checkpoints.
