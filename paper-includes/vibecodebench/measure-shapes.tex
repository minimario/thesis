VibeCodeBench was released in February 2025, a period when many non-technical
users built applications via ``vibe coding'': describing the product they wanted
in natural language and delegating the implementation to a model. While many
were excited by the concept, others were critical, critiquing the idea of a
model autonomously building a complete application without significant human
intervention. Critics argued that models were not yet capable of handling the
complexity of full application development and would produce insecure, buggy, or
unusable results.

While the idea was popular, there were no existing benchmarks that precisely measured
the ``zero-to-one'' process of building the application from scratch. One tricky aspect
of designing such a benchmark is that implementations of the same specification can vary
widely, making it hard to design unit tests that can cover the full range of correct implementations.
A key innovation of VibeCodeBench was therefore the use of browser-based workflows as an evaluation mechanism.
By defining user workflows that describe the intended behavior of the application, we used a
model-controlled browser evaluator to interact with the generated application and
judge whether it met the specification. In Section \ref{sec:vibecodebench-alignment}, we validated
this evaluation approach by showing that model judgments aligned quite well with human judgments of application behavior.

The broader impact of VibeCodeBench is that it pushes coding evaluations toward
the form that today's more capable models require. Instead of constraining models to
implementing isolated functions and small repository-level edits, benchmarks need to measure
end-to-end software construction under substantial design freedom. Just like how developers use
coding assistants today, the agent should be in charge of choosing an implementation strategy,
writing tests, running the code, and iteratively debugging until the application works. With applications
of such scale, the way that evaluation is done will also need to evolve. We expect both browser-workflow evaluation and
end-to-end, specification-driven tasks to become increasingly important as coding agents continuously improve.

\subsection{Benchmark Design and Evaluation Metrics}

VibeCodeBench contains 100 web-application specifications spanning
personal tools to enterprise applications. These specifications include
common production requirements such as authentication, databases, payments, and email.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{generation-schematic-2.jpg}
  \caption{VibeCodeBench generation flow from natural-language specification to
  runnable application artifact.}
  \label{fig:vibecodebench-generation-thesis}
\end{figure}

Figure~\ref{fig:vibecodebench-generation-thesis} illustrates the generation
setup. A model starts from a natural-language product specification and works
inside a five-hour development environment with an OpenHands-based harness,
browser and terminal tools, SQL access, and sandboxed services such as Supabase
and MailHog. The output is a runnable Docker artifact containing the generated
application.
Each specification is paired with browser workflows that describe what a user
should be able to do in the generated application. For example, a workflow for a
social media platform involves creating an account, creating a new post,
commenting on the post, and then signing out. Each workflow includes a brief
purpose statement and a set of substeps, which are individual actions such as
``Navigate to Page Y and click Button Z''. Across the benchmark, the 100 tasks
contain 964 workflows and 10,131 substeps.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{evaluation_schematic.jpg}
  \caption{VibeCodeBench evaluates generated applications through browser-based
  workflows, making end-to-end product behavior the object of measurement.}
  \label{fig:vibecodebench-eval-thesis}
\end{figure}

Figure~\ref{fig:vibecodebench-eval-thesis} shows the evaluation side of the
benchmark. After the Docker container returned by the agent is deployed, Browser Use executes the prewritten workflows in a
fresh browser session and returns substep-level judgments. This browser-based
evaluation is a core design choice: instead of relying on unit tests, the
evaluator interacts with the application as a user would and checks whether the
intended behavior is achieved. As a result, VibeCodeBench is
implementation-agnostic while still measuring functional correctness. A workflow
passes when at least 90\% of its substeps succeed, and an application's score is
the percentage of workflows that pass. The reported metrics are mean application
accuracy on the held-out test set, generation cost, and wall-clock latency.

The analysis identified a strong relationship between browser self-testing and
final application quality. Browser tool calls per application correlated with
accuracy (Pearson $r=0.72$), while edit volume was much less predictive. This
indicated that VibeCodeBench rewarded agents that build, run, inspect, and repair
their applications in the same medium used by the evaluator.

Error analysis showed that model improvements were driven in large part by
reductions in complete failures. Lower-performing models often produce
applications that do not start or cannot complete any workflow, while stronger
models more often produced partially working applications. External integrations
remained a major source of difficulty in a majority of cases.

\subsection{Human Alignment and Impact} \label{sec:vibecodebench-alignment}

One of the key innovations of VibeCodeBench was the model-controlled browser evaluator
workflow. In order to validate this new evaluation methodology, we conducted a
human-alignment study to measure how well automated judgments aligned with
human perceptions of application behavior.

\subsubsection{Methodology}

We selected six tasks covering a range of domains, difficulties, and
integrations. For each task, we generated an application using each of Gemini 3.1 Pro, GPT-5.2, and Claude Sonnet 4.6,
resulting in 18 applications and 1,401 unique substeps. Each application was evaluated by
two human evaluators and four model evaluators (Gemini 3.1 Pro, GPT-5.2, Claude Sonnet 4.6, and
Claude Sonnet 4.5). For each substep, the evaluator was asked to mark it passing or failing, and we
measured alignment on substeps as a binary judgement of whether two evaluators agreed.

Table~\ref{tab:vibecodebench-alignment-thesis} reports pairwise substep
alignment between all evaluator pairs. Human-human agreement was high, ranging
from 88.6\% to 93.6\%, indicating that manual judgments were relatively consistent.
Model-model agreement was more heterogeneous: Claude Sonnet 4.6 and Claude Sonnet 4.5 aligned strongly with each other
(87.8\%), Gemini 3.1 Pro was close to both Claude evaluators (86.4--86.5\%), and
GPT-5.2 had low agreement with the other model evaluators (33.1--38.8\%). Model-human agreement was generally high but varied, with a mean of 86.4\% for
Claude Sonnet 4.5, 86.3\% for Claude Sonnet 4.6, 84.7\% for Gemini 3.1 Pro,
and 36.1\% for GPT-5.2. Claude Sonnet 4.5 was therefore the strongest evaluator
in our experiments, so we used it as the primary model evaluator in the benchmark.

Our alignment experiment influenced how the benchmark was received by
practitioners. A researcher at a frontier lab reported that they had been
considering how to evaluate coding agents but were unsure whether automated
agent evaluation would be accurate. They emphasized that the
human-model alignment study made the benchmark more convincing and that several
findings from the paper reduced the number of validation experiments their team
needed to run. They described our workflow design as effective because it plays a
similar role to unit tests without making strong implementation-level assumptions about the
code.
